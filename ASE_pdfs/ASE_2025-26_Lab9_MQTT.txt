2. Introduction
The evolution of the Internet of Things (IoT) has motivated the search for efficient,
reliable, and low-power communication solutions, especially in embedded systems,
which operate with limited hardware, connectivity, and power resources. In this
scenario, the Message Queuing Telemetry Transport (MQTT) protocol has become
one of the main choices for real-time communication between connected devices.
MQTT is a lightweight messaging protocol based on the publisher-subscriber model,
originally developed by IBM for industrial applications over low-bandwidth and highlatency networks. These characteristics make it especially suitable for embedded
devices, such as the ESP32 family of microcontrollers, which frequently operate in
environments with limited or unstable connectivity, such as home or rural Wi-Fi
networks.
One of the main advantages of MQTT is its low communication overhead. Unlike
heavier protocols like HTTP, MQTT sends compact messages, optimized for minimal
data and power consumption, which is a crucial factor for battery-powered devices.

1

Furthermore, the protocol offers configurable Quality of Service (QoS) levels,
ensuring message delivery even in the case of temporary connection loss.
The publisher-subscriber model adopted by MQTT also provides strong decoupling
between message senders and receivers. This enables a more scalable, modular
architecture in which sensors, actuators, servers, and mobile applications can
interact asynchronously without direct dependencies on one another. Another
important benefit is support for features such as message retention and last will,
which make the system more robust and fault-tolerant.

2.1 MQTT Publisher-Subscriber Architecture
Figure 1 illustrates the publisher-subscriber architecture of MQTT. As can be
observed, an architecture is adopted where communication between devices is
mediated by a central element called a broker. The broker is responsible for
receiving client messages and distributing them only to subscribers of the
corresponding topic, ensuring an efficient, decoupled data flow.
For example, in a system where ESP32 devices monitor temperature, a client can
post values to the sensors/temperature topic, while other clients subscribed to that
topic receive updates automatically. This approach simplifies communication, avoids
direct connections between devices, and enables greater scalability and flexibility,
making it especially advantageous in embedded systems and IoT applications.

2.2 Quality of Service
The MQTT protocol defines three Quality of Service (QoS) levels that govern the
delivery guarantees between clients and the broker. These levels are especially
relevant in embedded systems, where network stability can vary and the balance
between reliability and efficiency is essential:
•

QoS 0 – “At most once”: The message is sent only once, without confirmation
of receipt. It is the fastest and lightest mode, but it does not guarantee that
the message will reach its destination.

•

QoS 1 – “At least once”: The message is guaranteed to be delivered, but may
be received more than once in case of retransmissions.

•

QoS 2 – “Exactly once”: Ensures that the message is delivered only once,
without duplication. It is the most secure level, but also the most resourceintensive and latency-intensive.

The choice of QoS level depends on the type of application: sensors with frequent
updates can use QoS 0 for efficiency, while critical commands may require QoS 1 or
QoS 2 for greater reliability. QoS is defined per message sent or per topic
subscription.

2.3 Topic
In the MQTT protocol, a topic is a hierarchical string used to organize and classify
messages that are published and subscribed to. It functions as a "channel" for
sending (publishing) or receiving (subscribing) data. Topics are fundamental to the
publisher/subscriber model, allowing devices to communicate in a decoupled way,
that is, without the sender and receiver knowing each other directly:
•

A publishing client sends messages to a specific topic.

•

A subscribing client receives messages when it subscribes to that topic.
3

The structure of a topic is similar to a directory path, separated by slashes (/):
•

Example: house/room/temperature

Each level represents a category or group of information, allowing the organization
of the data in a logical and scalable way.

2.4 How to Use MQTT on the ESP32-C6
The MQTT protocol can be used on the ESP32-C6 through different transport modes,
adapting to various security contexts, network infrastructure, and application
requirements:
•

MQTT over TCP (mqtt/tcp): The most basic form, using port 1883 and offering
no encryption layer. This configuration is only suitable for controlled and
reliable environments, such as local area networks (LANs) or VPNs, where
security is guaranteed by other means. It is useful for testing or prototyping,
allowing connection to a broker, simple topic publishing and subscription,
and reconnection management in case of Wi-Fi failures. However, because it
transmits credentials and data in plain text, it should not be used on open
networks or the Internet.

•

MQTT over TLS (mqtt/ssl): Recommended mode for production applications
or those with Internet exposure and operates on port 8883. This mode adds
end-to-end encryption via digital certificates, ensuring data confidentiality
and integrity. On the ESP32-C6, it is possible to embed Certificate Authority
(CA) certificates in the firmware to validate the server certificate and
optionally support mutual authentication with a client certificate. This model
is widely adopted in cloud-based solutions, such as AWS IoT or Azure,
although it requires additional care with the certificate lifecycle, memory
consumption, and reconnection and backoff policies.

•

TLS with pre-shared keys (mqtt/ssl_psk): A lighter variation that replaces the
certificates with a common symmetric key between the client and the broker.
This approach reduces cryptographic complexity and memory usage, making
4

it useful in closed environments, such as laboratories or private industrial
networks. However, secure management of shared keys is complex on a large
scale, and many commercial brokers do not support this method, making it
impractical for public applications.
•

MQTT over WebSocket (mqtt/ws): An alternative that encapsulates MQTT
traffic within an HTTP connection via WebSocket, typically on port 80. This
approach is useful when the network infrastructure blocks direct TCP
connections but allows HTTP traffic, such as in corporate networks with
restrictive policies. However, when used without TLS (ws://), communication
remains vulnerable. Therefore, in real-world scenarios, the secure version
with TLS (mqtt/wss) is recommended, which uses port 443 and combines the
advantages of WebSocket with TLS security.

•

Secure MQTT over WebSocket with TLS (mqtt/wss): Effective in business
contexts or public networks, where it is common for only the HTTPS port to
be accessible. In addition to ensuring secure communication, it allows the
MQTT broker to operate behind load balancers, HTTP proxies, or CDNs,
sharing infrastructure with other web services. This configuration is ideal for
IoT devices operating in environments with restrictive firewalls, and is often
adopted as the standard solution in modern IoT architectures.

2.5 MQTT Broker Options
There are several MQTT brokers available for local or server installation, each with
specific characteristics that adapt to different usage scenarios. One of the most
popular is Eclipse Mosquitto, a lightweight, open-source solution maintained by the
Eclipse Foundation. It is ideal for local testing, ESP32 applications, and small to
medium-scale projects. It supports MQTT versions 3.1, 3.1.1, and 5.0, as well as
features such as user and password authentication, TLS encryption, and message
persistence. Its installation is simple, with packages available for Windows, Linux,
macOS, and also via Docker.

5

For larger-scale applications and corporate environments, EMQX stands out as one
of the most complete options. This broker is highly scalable, supporting millions of
simultaneous connections, and includes a wide range of features, such as advanced
authentication (including OAuth2 and JWT), access control (ACL), MQTT v5 support,
integration with external databases and protocols, as well as a web interface for
monitoring and management. It can be installed locally or in Docker and Kubernetes
containers, and is also available in a cloud version.
Another robust alternative is HiveMQ CE (Community Edition), the free and opensource version of the HiveMQ broker. It focuses on reliability, performance, and
integration with business systems. It supports MQTT v3.1.1 and v5, and is extensible
through Java plugins, allowing it to easily adapt to different architectures.
Installation is also cross-platform, with support for modern operating systems and
deployment via Docker.
Finally, RabbitMQ, widely known as an AMQP broker, can also serve as an MQTT
broker by enabling the MQTT plugin. This functionality makes RabbitMQ particularly
useful in hybrid environments, where you want to integrate MQTT devices with
existing AMQP-based systems. With support for TLS, authentication, administrative
dashboards, and the ability to interconnect multiple protocols, RabbitMQ with MQTT
support is ideal for those already using this technology and wanting to expand to IoT
applications in an integrated way.

3.3 Broker Mosquitto
In this part of the script, you will need to install the Mosquitto broker and configure
it to operate in both MQTT over TCP and MQTT over TLS modes, making it
compatible with the examples explored previously.
Important note: The Mosquitto broker must be installed on a personal laptop with
direct connectivity to the ESP32-C6 board via the local Access Point (cell phone), i.e.,
they must be on the same network. Given that the Common Name (CN) of the CA
certificate must be equal to the broker's IP address (so that TLS authentication does
not fail), it is recommended that the laptop's connection to the AP be made at the
beginning and that the respective IP address assigned by the DNS server running on
the AP be registered. This will save a lot of work later.
The installation is simple, just run the following commands in the command line:

14

sudo apt-get update
sudo apt-get install mosquitto mosquitto-clients

Next, generate the certificate associated with the broker. Note that you must
replace the IP address in the example with the laptop's IP address where the broker
will run.
cd /etc/mosquitto/certs
# Replace 192.168.1.50 with the actual IP of your PC in the LAN
sudo openssl req -new -x509 -days 365 -nodes \
-out mosquitto.crt \
-keyout mosquitto.key \
-subj "/CN=192.168.1.50"
sudo chown mosquitto:mosquitto mosquitto.*
sudo chmod 600 mosquitto.key
sudo chmod 644 mosquitto.crt

Next, edit the file /etc/mosquitto/mosquitto.conf so that it contains the following:
pid_file /run/mosquitto/mosquitto.pid
user mosquitto
persistence true
persistence_location /var/lib/mosquitto/
log_dest syslog
log_dest stdout
log_type error warning notice information
include_dir /etc/mosquitto/conf.d

Create the file /etc/mosquitto/conf.d/local.conf with the following content:

15

# Plain MQTT, no auth
listener 1883
allow_anonymous true
# TLS MQTT, no auth
listener 8883
cafile /etc/mosquitto/certs/mosquitto.crt
certfile /etc/mosquitto/certs/mosquitto.crt
keyfile /etc/mosquitto/certs/mosquitto.key
allow_anonymous true

Next, launch the Mosquitto broker:
sudo systemctl start mosquito

To test if this is working correctly, try the following commands. Note: the IP address
with the corresponding address for your broker must be replaced.
# Plain
mosquitto_sub -h 192.168.1.50 -p 1883 -t test -v
# TLS
mosquitto_sub -h 192.168.1.50 -p 8883 \
--cafile /etc/mosquitto/certs/mosquitto.crt \
-t test -v

After performing these operations, you will have validated that the local Mosquitto
broker is functioning correctly.
