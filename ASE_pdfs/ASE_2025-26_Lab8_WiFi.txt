2. Introduction
Wi-Fi is a registered trademark of wireless communication technology owned by the
Wi-Fi Alliance (WFA), the entity responsible for overseeing the Wi-Fi certification
process. It is a family of wireless network protocols based on the IEEE 802.11
standards. Products that pass rigorous testing receive the Wi-Fi CERTIFIED™ seal,
proving they meet industry-defined requirements for interoperability, security, and
application-specific protocols. Compared to other wireless communication
technologies, Wi-Fi offers greater coverage, better penetration through walls, and
higher transmission rates.
The fundamental element of an IEEE 802.11 network is the Basic Service Set (BSS).
This can be divided into two categories: Standalone BSS and Infrastructure BSS. In a
BSS infrastructure, stations (STAs) must associate with an Access Point (AP) to obtain
network services. Access Points (APs) function as the control center for the cluster,
which is the most common network architecture. Each Access Support Terminal
(STA) must go through the association and authorization processes before joining a
given Business System Support System (BSS).

1

From a practical perspective, Wi-Fi network architectures are accompanied by their
own identifiers:
•

BSS Identification (BSSID) – each BSS has a physical identification address,
called a BSSID. In an Infrastructure BSS, the BSSID corresponds to the MAC
address of the Access Point (AP). This value is set by default, but can be
changed following a specific naming format.

•

Service Set Identification (SSID) – each AP also has an identifier intended for
the user, called an SSID. In most cases, a BSSID is associated with a single
SSID. This identifier is usually a human-readable text, commonly called the
Wi-Fi network name.

Given this scope, it is important to understand how these same structures and
identifiers apply to embedded devices that integrate wireless connectivity. In the
case of the ESP32-C6, the ESP-IDF framework provides full Wi-Fi support according to
IEEE 802.11 standards, allowing the SoC to function both as a Station (STA) connecting to an existing Access Point - and as a SoftAP, creating its own network.
The configuration of these functionalities is done through specific APIs that allow
defining the SSID, security, channel, and managing association events, IP address
acquisition, and loss of connectivity.
The integration of Wi-Fi in the ESP32-C6 follows exactly the principles described
above: when acting as an STA, the device goes through the process of network
search, authentication, and association with a BSSID; when acting as a SoftAP, the
ESP itself announces the SSID, provides IP addresses via DHCP, and manages the
stations that connect.

3.2 Wi-Fi “Soft” Access Point (SoftAP) Mode
In the SoftAP example, we will configure the ESP32-C6 to operate as a Wi-Fi Access
Point. This means that the ESP itself creates and advertises an autonomous Wi-Fi
network, allowing other devices (such as smartphones, laptops, or IoT modules) to
connect directly to it, without the need for an external router.
The example includes a monitoring mechanism that prints log messages whenever a
station (device) connects to the access point. In this context, the ESP records the
device's MAC address, association identifier (AID), and other relevant information,
allowing you to know exactly when and who joined the network.

SoftAP mode is useful at various stages of an IoT device's lifecycle. First, it facilitates
initial setup, allowing the user to connect directly to the device via their mobile
phone to enter Wi-Fi network credentials or adjust essential parameters. This mode
is equally advantageous in scenarios where the device needs to operate
autonomously, without relying on an external router, such as in temporary
installations, isolated industrial environments, or field measurements. In addition, it
can be a very practical tool for diagnostics and maintenance, enabling the recovery
of devices with connection failures, the changing of settings, or the performance of
tests without resorting to cables or additional infrastructure. Finally, SoftAP allows
providing local control interfaces, such as configuration web pages or dashboards,
ensuring direct communication without the need for internet access.

3.3 Web Server
In this section, we will cover the https_server/simple example of the ESP-IDF
platform for the ESP32-C6 SoC. This example demonstrates how to create a secure
web server (HTTPS) running directly on the device. The program establishes a
network connection (via Wi-Fi), loads a certificate and private key embedded in the
firmware, and starts an HTTPS server that listens for requests on the secure port.
Then, it registers a simple handler for the GET method on the route “/”, responding
with a small HTML page. The example also shows how to start and stop the server
automatically when the network interface gains or loses connectivity, illustrating the
basic operation of the TLS stack, the secure HTTP server, and network event
management on the ESP32-C6. For this exercise, you will need to configure your
mobile phone as an Access Point so that it assigns an IP address to the ESP32-C6 SoC
upon initialization.

Question 8: Connect your computer to the Access Point created by the cell phone
and access the address 192.168.68.58. As you can see, the browser will display a
security warning, preventing you from immediately viewing the page served by the
ESP32-C6.
This happens because the certificate used by the server is self-signed — that is, it
was not issued by a recognized certification authority (such as Let’s Encrypt or
DigiCert). Thus, the browser cannot confirm the server's identity and considers the
connection potentially insecure.
This factor is usually accompanied by another: the name indicated in the certificate
does not match the address used to access the device (for example, we are using
https://192.168.68.58/, but the certificate lists a different name). In a development
and testing context, this behavior is expected. The solution is simply to accept the
security exception in the browser or, alternatively, generate a certificate with the
correct name or install the issuing authority as trusted in the system.
For now, accept the security exception, and you should see the web page in Figure 2.
